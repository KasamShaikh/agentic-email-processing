ⓘ What this app is

An agentic email-processing pipeline on Azure. Emails land in an Office 365 inbox; a Foundry orchestrator agent reads each one, decides what kind of email it is, and routes it to a specialist agent that does the real work — extracting trades from a broker contract note, or cross-checking a merchant’s onboarding forms — and writes a clean, auditable output file back to storage.

This screen is the live operations view: the centre panel streams every step in real time, the routing map on the right shows which agent fired, and the Logic App panel lists the real email-triggered runs.

🧠 The model & platform

LLM: a GPT‑class small model deployed on Azure AI Foundry (deployment gpt-mini-ks), chosen for low latency and cost on high-volume mailbox traffic.

Agent runtime: Azure AI Foundry Agents. Each agent is a named, versioned assistant with its own system prompt and tools; the app resolves them by name at run time via the Azure AI Agents SDK.

OCR: Azure AI Document Intelligence (prebuilt-layout, markdown output) turns scanned PDFs and images into text the agents can read.

Auth: Microsoft Entra managed identity only — no keys. Blob shared-key access is disabled.

⚛ The core design principle

The LLM proposes; deterministic code decides.
Agents do the fuzzy, language-heavy work — reading documents, aligning fields that are worded differently. Plain Python then makes the final, repeatable decision (scoring, formatting, thresholds).

Example: in onboarding, form-compare-ks aligns the two forms semantically, but a deterministic scorer applies fixed thresholds (≥0.95 = match, ≥0.80 = near, else mismatch) so the same forms always yield the same verdict — essential for an auditable process.

➤ How a message flows, end to end

  1. Email arrives A new mail hits the Office 365 inbox and triggers the logic-email-ks Logic App.
  2. Ingest & stage The Logic App saves each attachment to the incoming-attachments blob container and composes a JSON payload (subject, body, blob paths).
  3. Classify intent The orchestrator agent (gpt-mini-ks) reads the payload and classifies it: contract_note, pre_onboarding or manual.
  4. Route to a specialist Code-driven routing hands the work to the matching agent — the model decides the intent, the app decides the wiring.
  5. Do the work  • Contract note: OCR → extract trades → resolve identifiers → build the pipe-delimited file.
     • Onboarding (demo): OCR both forms → agent aligns fields → deterministic scorer computes match % and flags mismatches.
  6. Write output The result lands in the contract-notes-output container (a .txt report + machine .json).
  7. Observe Every step is streamed to this dashboard over Server-Sent Events and saved as a replayable trace.

🗂 Inputs, outputs & storage

Input: incoming-attachments — raw email attachments.

Demo forms: onboarding — the two fixed sample forms used for the onboarding comparison.

Output: contract-notes-output — generated upload files and onboarding reports.

All access is via managed identity; nothing is read from or written with account keys.
