ⓘ What this app is

An agentic email-processing pipeline on Azure. Emails land in an Office 365 inbox; an orchestrator agent reads each one, decides what kind of email it is, and routes it to a specialist agent that does the real work — extracting trades from a broker contract note, or cross-checking a merchant’s onboarding forms — and writes a clean, auditable output file back to storage.

This screen is the live operations view: the centre panel streams every step in real time, the routing map on the right shows which agent fired, and the Logic App panel lists the real email-triggered runs.

Version 2 · Microsoft Agent Framework (MAF). Functionally identical to v1, but the agent runtime is now the open-source Microsoft Agent Framework running in-process instead of the Azure AI Agents SDK and the Foundry-hosted agent runtime. The model (gpt-mini-ks), Logic App ingestion, Document Intelligence, Blob storage and every output file are unchanged. Agents are now defined in code and their tools are invoked automatically by the framework — there is no server-side threads/runs object and no manual tool-output plumbing. This is a separate copy of the app; the original v1 deployment and its Azure resources are left exactly as they are.

▤ High-level architecture

Email in → Logic App stages attachments to Blob → an in-process Microsoft Agent Framework workflow runs: the orchestrator (gpt-mini-ks) classifies the intent, a human-in-the-loop gate pauses for approval, then switch-case routing hands off to the specialist executor that runs the deterministic pipeline → the result is written back to Blob. Every call uses Entra managed identity — no keys.

🧠 The model & platform

LLM: a GPT‑class small model deployed on Azure AI Foundry (deployment gpt-mini-ks), chosen for low latency and cost on high-volume mailbox traffic. Unchanged from v1.

Agent runtime (v2): the open-source Microsoft Agent Framework. Agents are defined in code as in-process Agent objects with their own instructions and tools, wired into a true MAF Workflow graph (executors + switch-case edges). The framework invokes real tools automatically (e.g. the contract agent's lookup_isin), and the workflow pauses natively for human approval via request_info, checkpointing each superstep to file-backed storage (FileCheckpointStorage). The model is still served by Foundry gpt-mini-ks.

OCR: Azure AI Document Intelligence (prebuilt-layout, markdown output) turns scanned PDFs and images into text the agents can read.

Auth: Microsoft Entra managed identity only — no keys. Blob shared-key access is disabled.

⚛ The core design principle

The LLM proposes; deterministic code decides.
Agents do the fuzzy, language-heavy work — reading documents, aligning fields that are worded differently. Plain Python then makes the final, repeatable decision (scoring, formatting, thresholds).

Example: in onboarding, form-compare-ks aligns the two forms semantically, but a deterministic scorer applies fixed thresholds (≥0.95 = match, ≥0.80 = near, else mismatch) so the same forms always yield the same verdict — essential for an auditable process.

➤ How a message flows, end to end

  1. Email arrives A new mail hits the Office 365 inbox and triggers the logic-email-ks Logic App.
  2. Ingest & stage The Logic App saves each attachment to the incoming-attachments blob container and composes a JSON payload (subject, body, blob paths).
  3. Classify intent The in-process Microsoft Agent Framework orchestrator executor (gpt-mini-ks) reads the payload and classifies it: contract_note, pre_onboarding or manual.
  4. Human-in-the-loop gate The workflow pauses natively (request_info) at one approval gate, its level set by the intent (🔴 L1 contract note · 🟠 L2 onboarding · 🟢 L3 manual). A reviewer can approve, reject or re-route from this dashboard; the run is held open — and checkpointed to file-backed storage each step — until a human decides, then auto-decides after the review window (≈15 min in the UI) so nothing hangs.
  5. Route to a specialist On approval, switch-case edges hand the work to the matching specialist executor — the model decides the intent, the workflow graph decides the wiring.
  6. Do the work  • Contract note: OCR → extract trades → resolve identifiers → build the pipe-delimited file.
     • Onboarding (demo): OCR both forms → agent aligns fields → deterministic scorer computes match % and flags mismatches.
  7. Write output The result lands in the contract-notes-output container (a .txt report + machine .json).
  8. Observe Every step is streamed to this dashboard over Server-Sent Events and saved as a replayable trace.

🗂 Inputs, outputs & storage

Input: incoming-attachments — raw email attachments.

Demo forms: onboarding — the two fixed sample forms used for the onboarding comparison.

Output: contract-notes-output — generated upload files and onboarding reports.

All access is via managed identity; nothing is read from or written with account keys.

🧑‍⚖️ How human-in-the-loop works (for business)

Read it left-to-right: the AI prepares every item, then routes by risk to one gate. 🟢 Exception check (manual email) pulls in a human for odd cases. 🟠 Maker review (onboarding) holds the run for a human approve/reject, auto-approving if the review window lapses. 🔴 Final sign-off (contract note) is a quick yes/no before the one irreversible write. Speed where it's safe, control where it matters — every decision is logged for audit.
